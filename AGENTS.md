# AGENTS.md

## 役割

このリポジトリは `dotnet-backend-snippets` です。C# / .NET のバックエンド開発で使うスニペットを、実装・テスト・説明ドキュメントのセットで管理します。

Codex は、単なるメモではなく **テストで動作確認できるスニペット集** として保守してください。

## 作業前の確認

- 既存のディレクトリ構成、命名規則、README / docs の書き方を先に確認する。
- 既存パターンに合わせ、不要な抽象化や大きなリファクタリングは避ける。
- 変更範囲は依頼内容に必要な最小限にする。

## スニペット追加・変更の基本

新しいスニペットは、原則として以下をセットで追加します。まだ対象ディレクトリがない場合は、この配置に沿って作成してください。

1. 実装コード: `src/DotnetBackendSnippets/<Category>/`
2. テストコード: `tests/DotnetBackendSnippets.Tests/<Category>/`
3. 説明ドキュメント: `docs/snippets/<category>/`

既存スニペットを変更した場合も、対応するテストとドキュメントの更新を検討してください。

## 実装方針

- C# の標準的な命名規則に従う。型・メンバーは PascalCase、ローカル変数と引数は camelCase。
- 非同期メソッドには `Async` サフィックスを付ける。
- `var` は型が明らかな場合に使う。
- サンプル用途でも、実務のバックエンド開発で再利用しやすい形にする。
- 例外処理、null、非同期処理、DI、設定、ログは実運用を意識する。
- 外部サービスに依存する処理は、インターフェースやテスト用実装を使って検証可能にする。
- 警告、不要な `using`、秘密情報の直書きを残さない。
- OS 依存の挙動を避ける。パス区切り文字、ファイル名禁止文字、改行、タイムゾーン ID、カルチャ、大小文字比較などは Windows / Linux / macOS で結果が変わらないよう明示的に正規化し、必要に応じてテストで固定する。

## テスト方針

- スニペットには対応するテストを追加する。
- 通常のロジックは単体テストで確認する。
- ASP.NET Core Web API は、必要に応じて結合テストを使う。
- EF Core は InMemory provider または SQLite in-memory を検討する。
- HttpClient はテスト用 `HttpMessageHandler` などを使い、外部通信を避ける。
- DI / Configuration は `ServiceCollection` / `ConfigurationBuilder` で検証する。
- テスト名は、何を確認しているか分かる名前にする。

## ドキュメント方針

日本語で簡潔に書き、専門用語には短い補足を入れてください。各スニペットの説明ドキュメントは、できるだけ次の構成にします。

```markdown
# スニペット名

## 目的
## 実装
## テスト
## 使い方
## メモ
```

`実装` と `テスト` には対象ファイルへの相対パスを記載してください。

## 確認コマンド

ソリューションまたはプロジェクトがある場合、変更後は可能な範囲で実行します。

```bash
dotnet restore
dotnet build
dotnet test
```

特定のテストプロジェクトだけ確認する場合:

```bash
dotnet test tests/DotnetBackendSnippets.Tests/DotnetBackendSnippets.Tests.csproj
```

実行できない場合は、理由と代替確認内容を最終報告に書いてください。

## 優先順位

迷った場合は、次の順に優先します。

1. テストで動作確認できること
2. 実務で再利用しやすいこと
3. 説明が分かりやすいこと
4. 既存構成と一貫していること
5. 将来メンテナンスしやすいこと
